# 执行计划

## 概述

加载计划，批判性复核，按 Stage — Task Group — Task — Step 层级执行，完成后报告。Step 包含的是需求描述而非可直接复制的代码，执行时需要根据规格自行编写实现。

**开始时声明：** "我正在使用 executing-plans 技能实现此计划。"

## 执行流程总览

```
加载完整计划 → 探索工作区 → 批判性复核 → 执行循环 → 全局验证 → 汇报结果
```

执行循环（对每个 Task 重复）：

```
读取进度清单，定位下一个未完成 Task
→ 读取该 Task 完整 Step 描述
→ 实现
→ 单 Task 验证
→ 提交
→ 更新进度清单标记完成
```

每次循环开头必须重新读取进度清单文件——这是上下文压缩后重新定位的唯一可靠锚点。

## 过程

### 步骤 1: 加载并复核计划

1. 检查计划目录下是否存在索引文件 `YYYY-MM-DD-<功能名称>.md`
   - 若存在：读取索引文件，了解所有 Stage 的范围和依赖关系，从 Stage 1 开始加载
   - 若不存在：直接读取单一计划文件
2. 读取当前 Stage 的计划文件
3. 理解分层结构：Stage（可选）→ Task Group（N，串行）→ Task（N-M，同 Group 并行）→ Step（串行）
4. 批判性复核——识别关于计划的任何问题或疑虑
   - 进度清单是否完整覆盖了所有 Task
   - 同 Group 内的 Task 是否文件层面互斥
   - Step 描述是否足够明确可执行
5. 如果有疑虑：在开始之前向用户提出
6. 如果没有疑虑：按 Group 顺序依次执行，每个 Group 内的 Task 完成后推进到下一个 Group

### 步骤 2: 执行 Task

对于每个 Group 按串行顺序处理，Group 内各 Task 也按串行顺序逐个执行：

1. 在进度清单中将该 Task 标记为进行中
2. 阅读每个 Step 的需求描述
3. 根据 Step 的规格要求自行编写实现代码和测试
4. 自行构造并运行验证命令，确认测试通过
5. 提交代码
6. 在进度清单中将该 Task 标记为已完成
7. 继续执行 Group 内下一个 Task，Group 全部完成后进入下一个 Group

### 步骤 3: 当前 Stage 完成

当前 Stage 全部 Task 完成并验证后：
- 运行与所做改动相匹配的验证命令
- 存在下一个 Stage 时，回到步骤 1 加载下一个 Stage
- 所有 Stage 完成后，向用户报告执行结果：完成了什么、测试结果、任何需要注意的问题

## 何时停止并寻求帮助

**立即停止执行当：**
- 遇到阻塞（缺失依赖、测试失败）
- 计划有关键缺口导致无法开始
- Step 描述过于模糊，关键细节缺失以至于无法确定应如何实现
- 验证反复失败

**寻求澄清而非猜测。**

## 何时重新审视前面的步骤

**返回复核（步骤 1）当：**
- 用户根据你的反馈更新了计划
- 基本方案需要重新思考

**不要强行突破阻塞**——停下来并询问。

## 记住
- 首先批判性复核计划
- Step 包含需求描述，执行时需要自行编写代码和命令
- 进度通过文档头部的清单跟踪，而非 Task 正文中的 checkbox
- 每完成一个 Task，必须立即更新进度清单，再继续执行下一个 Task
- 不要跳过验证
- 按计划要求参考相关技能
- 遇到阻塞时停下来，不要猜测
