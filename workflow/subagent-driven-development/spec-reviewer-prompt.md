# Spec 合规审查者提示词模板

在分派 spec 合规审查者子代理时使用此模板。

**目的：** 验证实现者构建了所要求的内容（不多不少）

```
分派 subagent（通用类型）：
  description: "审查 Task N 的 spec 合规性"
  prompt: |
    你正在审查某个实现是否与其规格说明匹配。

    ## 要求内容

    [任务需求的完整文本]

    ## 设计文档参考

    [设计文档路径或与当前 Task 相关的关键章节（架构、接口定义、错误处理策略），供交叉验证单 Task 实现与全局设计的一致性]

    ## 实现者声称已构建的内容

    [来自实现者的报告]

    ## 关键：不要信任报告

    实现者完成得异常迅速。他们的报告可能不完整、不准确或过于乐观。你必须独立验证一切。

    **不要：**
    - 相信他们所说的实现内容
    - 信任他们关于完整性的声明
    - 接受他们对需求的解读

    **务必：**
    - 阅读他们编写的实际代码
    - 将实际实现与需求逐行对比
    - 检查他们声称已实现但实际缺失的部分
    - 查找他们未提及的额外功能

    ## 你的任务

    阅读实现代码并验证：

    **缺失的需求：**
    - 他们是否实现了所有要求的内容？
    - 是否有跳过或遗漏的需求？
    - 他们是否声称某项功能可用但实际并未实现？

    **多余/不必要的工作：**
    - 他们是否构建了未被要求的内容？
    - 他们是否过度工程或添加了不必要的功能？
    - 他们是否添加了 spec 中不存在的"锦上添花"功能？

    **误解：**
    - 他们是否以不同于预期的方式解读了需求？
    - 他们是否解决了错误的问题？
    - 他们是否实现了正确的功能但用错了方式？

    **通过阅读代码来验证，而非信任报告。**

    报告：
    - ✅ Spec 合规（如果代码检查后一切匹配）
    - ❌ 发现问题：[具体列出缺失或多余的内容，附文件:行号引用]
```
