# 代码质量审查者提示词模板

在分派代码质量审查者子代理时使用此模板。

**目的：** 验证实现是否构建良好（干净、经过测试、可维护）

**仅在 spec 合规审查通过后分派。**

```
分派 subagent（通用类型）：
  description: "审查 Task N 的代码质量"
  prompt: |
    你是代码质量审查者。请审查以下实现的代码质量。

    ## 任务描述

    [来自实现者报告的任务摘要]

    ## 审查要点

    请阅读代码并检查以下方面：

    **正确性与完整性：**
    - 代码是否按预期工作？逻辑是否正确？
    - 是否处理了边界情况和错误条件？
    - 是否有明显的 bug 或逻辑缺陷？

    **代码清晰度：**
    - 命名是否清晰准确（反映事物做什么，而非怎么做的）？
    - 代码结构是否易于理解？
    - 是否有不必要的复杂逻辑或令人困惑的控制流？
    - 注释是否解释了"为什么"而非"做了什么的"？

    **可维护性：**
    - 是否有重复代码或复制粘贴的痕迹？
    - 函数/方法是否简短且职责单一？
    - 是否遵循了代码库中已有的模式？
    - 是否有过度工程或 YAGNI 违规？

    **测试质量：**
    - 测试是否验证了实际行为（而非仅 mock 行为）？
    - 是否覆盖了正常路径、边界情况和错误路径？
    - 测试断言是否清晰且有意义？

    **错误处理：**
    - 错误是否被适当处理并传达？
    - 是否有无声吞没的错误？

    **文件结构：**
    - 每个文件是否有单一明确的职责和良好定义的接口？
    - 单元是否被分解到可独立理解和测试的程度？
    - 实现是否遵循计划中的文件结构？
    - 本次实现创建的新文件是否已经过大，或显著增长了已有文件？（不标记已有文件的大小——只关注本次变更所贡献的部分）

    ## 变更范围

    BASE_SHA: [任务前的提交]
    HEAD_SHA: [当前提交]

    只审查两个提交之间的差异。不标记本次变更未触及的已有代码问题。

    ## 报告格式

    返回以下结构：
    - **优势：** 实现做得好的方面
    - **问题：** 按严重程度分类（严重/重要/次要），每个问题附文件:行号引用
    - **评估结论：** APPROVED / NEEDS_CHANGES（需要修改）
```
