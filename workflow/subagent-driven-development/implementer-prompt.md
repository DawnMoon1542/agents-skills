# 实现者子代理提示词模板

在分派实现者子代理时使用此模板。

```
分派 subagent（通用类型）：
  description: "实现 Task N：[任务名称]"
  prompt: |
    你正在实现 Task N：[任务名称]

    ## Task 描述

    [计划中 Task 的完整文本——粘贴到此处，不要让子代理自行读取文件]

    ## 上下文

    [场景设定：此 Task 在整个计划中的位置、依赖关系、架构上下文]

    ## 开始之前

    如果你对以下内容有疑问：
    - 需求或验收标准
    - 方法或实现策略
    - 依赖或假设
    - Task 描述中任何不清楚的地方

    **现在提出。** 在开始工作之前提出任何顾虑。

    ## 你的工作

    一旦需求明确：
    1. 严格按 Task 指定的内容实现
    2. 编写测试（如果 Task 要求，遵循 TDD）
    3. 验证实现是否可用
    4. 提交你的工作
    5. 自审（见下文）
    6. 报告结果

    工作目录：[目录]

    **工作中：** 如果遇到意外或不清楚的情况，**提出问题**。随时可以暂停并澄清。不要猜测或做假设。

    ## 代码组织

    你最能理解可以一次性放入上下文的代码，当文件聚焦时你的编辑也更可靠。请记住：
    - 遵循计划中定义的文件结构
    - 每个文件应有单一明确的职责和良好定义的接口
    - 如果你正在创建的文件超出计划的预期范围，停止并将其报告为 DONE_WITH_CONCERNS——不要在无计划指导的情况下自行拆分文件
    - 如果你正在修改的已有文件已经很大或混乱，小心操作并在报告中将其记录为顾虑
    - 在已有代码库中，遵循已有模式。像优秀开发者那样改进你正在接触的代码，但不要重构 Task 范围之外的内容

    ## 当你力不能及

    随时可以停下来并说"这对我来说太难了"。糟糕的工作比没有工作更糟。你不会因升级问题而受到惩罚。

    **在以下情况停止并升级：**
    - Task 涉及有多种合理方案的架构决策
    - 你需要理解超出给定范围的代码并且无法找到清晰答案
    - 你对自己的方案是否正确感到不确定
    - Task 涉及以计划未预料的方式重构已有代码
    - 你一直在逐个文件阅读试图理解系统但毫无进展

    **如何升级：** 以 BLOCKED 或 NEEDS_CONTEXT 状态报告。具体描述你卡在哪里、你尝试了什么、以及你需要什么帮助。控制器可以提供更多上下文、使用更强的模型重新分派，或将 Task 拆分为更小的部分。

    ## 报告前：自审

    以全新的眼光审查你的工作。问自己：

    **完整性：**
    - 我是否完整实现了 spec 中的所有内容？
    - 我是否遗漏了任何需求？
    - 是否有我未处理的边界情况？

    **质量：**
    - 这是我最好的工作吗？
    - 命名是否清晰准确（反映事物做什么，而非怎么做的）？
    - 代码是否干净且可维护？

    **纪律：**
    - 我是否避免了过度构建（YAGNI）？
    - 我是否只构建了被要求的内容？
    - 我是否遵循了代码库中已有的模式？

    **测试：**
    - 测试是否验证了实际行为（而非仅 mock 行为）？
    - 如果要求 TDD，我是否遵循了？
    - 测试是否全面？

    如果在自审中发现问题，在报告之前立即修复。

    ## 报告格式

    完成后报告：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 你实现了什么（或如果被阻塞，你尝试了什么）
    - 你测试了什么以及测试结果
    - 变更的文件
    - 自审发现（如果有）
    - 任何问题或顾虑

    如果你完成了工作但对正确性有疑虑，使用 DONE_WITH_CONCERNS。
    如果你无法完成 Task，使用 BLOCKED。如果你需要未提供的信息，使用 NEEDS_CONTEXT。
    绝不要默默交出你不确定的工作。
```
