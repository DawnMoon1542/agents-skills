# 编写实现计划

## 概述

接收 grill-with-docs 产出的设计文档（含架构、接口定义、错误处理、测试策略），将其转化为全面的实现计划。假设执行者对我们的代码库零上下文、品味存疑。记录他们需要知道的一切：每个 Task 要触及哪些文件、代码、测试、可能需要查阅的文档、如何测试。以小块 Task 的形式给出完整计划。DRY。YAGNI。TDD。频繁提交。

假设他们是熟练的开发者，但几乎不了解我们的工具集或问题领域。假设他们不太了解良好的测试设计。

**输入：** grill-with-docs 产出的设计文档（`docs/grill/YYYY-MM-DD-<slug>.md`，具体路径由上游 grill-with-docs 在交接时提供），以及已更新的 `docs/CONTEXT.md` 术语表。编写计划时使用 CONTEXT.md 中定义的规范术语，保持术语一致性。

**开始时声明：** "我正在使用 writing-plans 技能编写实现计划。"

**上下文：** 如果在隔离工作区中工作，执行时应确保环境准备妥当。

**保存计划到：** 项目约定的计划文档位置；如无约定，保存到 `docs/plans/YYYY-MM-DD-<slug>.md`（无 Stage）或 `docs/plans/YYYY-MM-DD-<slug>-stage-N.md`（有 Stage）。沿用 brainstorming 确定的 slug。
- （用户偏好设置覆盖此默认值）

## 分层结构：Task Group — Task — Step

计划按三层组织。层级越高越粗，越低越细。

> **关于 Stage：** Stage 是 brainstorming 层级的概念，用于将过于复杂的需求分割为多个里程碑，每个 Stage 独立经历完整的 brainstorming→grill→plan→execute 循环。writing-plans 每次只处理一个 Stage（或无 Stage 的完整需求），不在计划层级引入 Stage 拆分。如果收到的设计文档范围仍然过大，应回退要求上游 brainstorming 重新分割 Stage，而非自行拆分。

### Task Group（N）

Task Group 是将**可独立交付的工作**分组的方式。Group 之间串行：Group 2 在 Group 1 所有 Task 完成后才能启动。

### Task（N-M）

每个 Task 对应一个独立可完成的单元工作。命名规则：

- `Task 1-3`：Group 1 的第 3 个 Task
- `Task 2`：Group 2 的唯一 Task（未划分 M 时省略 M）

同一个 Group 内不同 Task 之间在逻辑上可以并行执行，但必须满足**文件层面互斥**——同 Group 内的任意两个 Task 不得修改同一个文件。若两个 Task 都需要修改同一文件，将其合并为一个 Task，或拆分到不同的 Group 中串行执行。

> **当前执行说明：** executing-plans 和 subagent-driven-development 当前均以串行方式执行同 Group 内的 Task。N-M 分组格式为未来并行优化保留了分组依赖信息，当前使用中无需刻意并行。

### Step

一个 Task 内的 Step 严格串行，按编号顺序执行。每个 Step 用文字详细描述应完成的工作和要求，不直接产出代码或命令。

## 范围检查

writing-plans 每次只处理一个 Stage 的设计文档。如果收到的设计文档覆盖多个独立子系统且规模过大，不应自行拆分，而应向用户说明问题并建议回退到 brainstorming 阶段重新分割 Stage。

以下信号提示设计文档规模可能过大：
- 设计文档包含 3 个以上相互独立的子系统
- 预估 Task 数量超过 15 个
- 预估计划文档超过 300 行

## 文件结构

在定义 Task 之前，规划哪些文件将被创建或修改，以及每个文件负责什么。

- 设计具有清晰边界和良好定义接口的单元。每个文件应有明确的单一职责。
- 优先选择更小、更专注的文件，而不是做太多事情的大文件。
- 经常一起修改的文件应该放在一起。按职责拆分，而非按技术层拆分。
- 在现有代码库中，遵循既定模式。如果代码库使用大文件，不要单方面重组——但如果你正在修改的文件已经变得臃肿，在计划中包含拆分是合理的。
- 验证并行安全：检查同 Group 内的任意两个 Task 是否修改同一文件。若冲突，合并 Task 或调整 Group 划分。

## Step 粒度

每个 Step 是一个动作（2-5 分钟可完成），用文字详细描述：

- 应做什么操作（修改哪个文件、新增哪个文件）
- 具体要求是什么（函数签名、边界条件、错误信息文字）
- 预期结果是什么（测试通过的表现、运行后的输出变化）

不要直接写出大段完整代码或命令。Step 描述的是规格要求，代码和命令在执行阶段才产出。当一小段代码比文字描述更精简且更准确时（如函数签名、接口定义、数据结构），允许使用小段代码来表述需求。

## 计划文档头部

**每个计划必须以以下头部开始：**

```markdown
# [功能名称] 实现计划

> **执行者须知：** 推荐使用 subagent-driven-development 技能逐 Task 实现；也可在当前会话中直接执行。

**目标：** [一句话描述构建内容]

**架构：** [2-3 句话描述方案]

**技术栈：** [关键技术/库]

**设计文档：** `docs/grill/YYYY-MM-DD-<slug>.md`

**术语表：** `docs/CONTEXT.md`

---

## 进度清单

### Group 1
- [ ] Task 1-1 — [Task 简要描述]
- [ ] Task 1-2 — [Task 简要描述]

### Group 2
- [ ] Task 2-1 — [Task 简要描述]

---
```

进度清单汇总全部 Task 及其完成状态，便于快速查看整体进度。

## Task 结构

```markdown
### Task 1-1: [Task 名称]

**依赖：** 无 / Task 1-1 完成
**文件：**
- 创建: `exact/path/to/file.py`
- 修改: `exact/path/to/existing.py`
- 测试: `tests/exact/path/to/test.py`

#### Step 1: [步骤名称]

对 `exact/path/to/file.py` 进行 [具体操作]，要求：
- [具体要求 1]
- [具体要求 2]
- [边界条件或错误信息的具体文字]

预期结果：[测试通过后的表现或运行后的输出变化]

#### Step 2: [步骤名称]

对 `tests/exact/path/to/test.py` 新增测试函数，覆盖以下场景：
- [场景 1 及预期断言]
- [场景 2 及预期断言]

运行测试验证失败（函数或类型尚未定义）。

#### Step 3: 提交

提交信息格式：`feat: [简要描述本次 Task 完成的内容]`

提交范围：`tests/path/ src/path/`
```

Step 之间串行执行。每个 Step 只描述应做什么、要求是什么、预期结果是什么。不写出完整代码或精确命令。

## 禁止占位符

每个 Step 必须包含执行者需要的实际内容。以下是计划失败——永远不要写这些：
- 模糊描述："添加适当的错误处理"、"添加验证"、"处理边界情况"——必须写明处理什么错误、验证什么条件、边界值是什么
- 空 Step："为以上编写测试"——必须写明测试文件路径、测试函数名称、覆盖的场景列表
- "与 Task X-Y 类似"——每个 Task 必须独立可读，不依赖其他 Task 的上下文
- 引用未在任何 Task 中定义的类型、函数或方法名称
- "TBD"、"TODO"、"稍后实现"、"填写细节"

## 记住
- 始终使用精确的文件路径
- 每个 Step 用文字详细描述应做什么、要求是什么，不直接写出代码或命令
- 同 Group 内的 Task 必须文件层面互斥
- 进度清单汇总全部 Task，Task 正文中不重复放置 checkbox
- DRY、YAGNI、TDD、频繁提交

## 自审

在编写完整计划后，使用 [plan-document-reviewer-prompt.md](./plan-document-reviewer-prompt.md) 中的检查项，并对照设计文档检查计划。

内联自审是编写过程中的快速检查，捕捉明显问题。plan-document-reviewer-prompt.md 是完整的审查清单，用于自审通过后的最终验证。两者互补使用：先内联自审，再用 reviewer prompt 全面检查。

**1. 设计覆盖：** 浏览设计中的每个章节/需求。能否指出哪个 Task 实现它。列出任何缺口。

**2. 占位符扫描：** 检查每个 Step 是否包含具体的文件路径、函数名称、测试场景列表和边界值。上述「禁止占位符」部分中的任何模式都应修正。

**3. 命名一致性：** 后续 Task 中引用的类型、函数名、文件路径是否与前面 Task 中定义的一致。Task 3 中叫 `clearLayers()` 但在 Task 7 中叫 `clearFullLayers()` 是一个 bug。

**4. 并行安全检查：** 检查同 Group 内的所有 Task，确认任意两个 Task 不修改同一文件。若发现冲突，合并 Task 或调整 Group 划分。

如果发现问题，直接修正。如果发现设计需求没有对应的 Task，添加该 Task。

## 执行交接

保存计划后，提供执行选择：

**"计划文档已保存到 `docs/plans/YYYY-MM-DD-<slug>.md`。两种执行选项：**

**1. Subagent 驱动（推荐）** — 每个 Task 全新 subagent + 两阶段审查（spec 合规 + 代码质量）。Task 之间不暂停，仅在无法解决的阻塞时停止。适合无需人工逐步介入的场景。

**2. 会话内执行** — 在当前会话直接执行。遇到模糊描述或阻塞时停止寻求帮助。不含独立审查环节，依赖用户在场监督。适合需要人工随时调整方向的场景。

**选择哪种方式？"

如果用户在之前的 Stage 中已选择执行模式，沿用该选择，不再重复询问。**

**如果选择 Subagent 驱动：**
- 调用 subagent-driven-development 技能
- 每个 Task 全新 subagent + 两阶段审查

**如果选择会话内执行：**
- 调用 executing-plans 技能
- 在当前会话执行，遇阻塞时停下询问
